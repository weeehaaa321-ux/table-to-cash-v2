Phase 11 · Cron fix

Discovered while pulling v1's env: CRON_SECRET in v1 production is empty. The handlers in /api/cron/{shift-reminder,table-check} reject empty secrets, so every Vercel cron invocation has been 401-ing since launch. Both background jobs (table-idle alerts every minute, shift-end reminders every 30 minutes) have been silently dead in production.

Fix on v2

  • Generated a fresh 32-byte hex CRON_SECRET
  • Set it on v2's Vercel production env (encrypted)
  • Redeployed v2 — cron handlers now validate Vercel's Authorization: Bearer ${CRON_SECRET} header successfully
  • Background jobs now run on v2: table-check writes alerts and pushes; shift-reminder fires shift-end notifications

Fix on v1

Couldn't simply set the same secret on v1 — v1 and v2 share the production database, so dual-firing crons would write duplicate alerts and send duplicate pushes every minute. Instead, removed cron config from v1's vercel.json entirely:

  • Stashed unrelated WIP edits in v1 working tree (track/page.tsx, ImmersiveMenu.tsx, ChangeTableModal.tsx)
  • Set "crons": [] in E:\table-to-cash\vercel.json
  • Committed + pushed to main on the v1 repo (weeehaaa321-ux/ttc) — auto-deploy picked it up
  • Restored stashed WIP edits

Net: v2 owns cron work; v1 stops scheduling broken invocations. No double-fire risk possible. The two formerly-dead features (table-idle alerts, shift-end reminders) are now alive for the first time in production.
